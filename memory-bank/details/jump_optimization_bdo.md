# Branch Displacement Optimization (BDO)

rakusk におけるジャンプ命令のサイズ最適化（BDO）の実装詳細について記録します。この最適化は、アセンブル結果のバイナリサイズを最小化するために不可欠な職人芸的要素を含んでいます。

## 1. 概要

x86 のジャンプ命令（JMP, Jcc, CALL）には、相対アドレスの距離に応じて 2バイト（8ビット変位）から 5〜6バイト（32ビット変位）までの複数のバリアントが存在します。BDO は、これらを可能な限り最短のバリアントに自動選択する仕組みです。

## 2. アルゴリズム: マルチパス・固定点反復

前方参照（まだアドレスが確定していないラベルへのジャンプ）を解決するため、Pass 1 を複数回繰り返します。

1.  **楽観的初期化**: 初回パスの開始前に、すべての相対ジャンプ命令を最短（2バイト）と仮定します。
2.  **サイズ計算**: 各パスで、現在の PC（プログラムカウンタ）とラベルアドレスの差分を計算し、必要に応じて命令サイズを拡張します。
3.  **収束判定**: 命令サイズに変更がなくなるまで Pass 1 を繰り返します（通常 2〜3 回で収束しますが、複雑な「ドミノ倒し」ケースでは回数が増えます）。

## 3. 実装の重要ポイント（職人芸的要素）

### 3.1. 初期化対象の限定
すべての `JMP` 命令を一律に 2バイトにリセットしてはいけません。以下の命令はリセット対象外です：
- **間接ジャンプ**: `JMP EAX`, `JMP [mem]` など。これらは相対距離に関係なく、常に固定のエンコード（通常 2〜4バイト）を持ちます。
- **FAR ジャンプ**: `JMP FAR [mem]` など。これらは最低でも 4バイト以上を必要とし、`rel8` (2バイト) にはなり得ません。

誤ってこれらを 2バイトにリセットすると、Pass 2 で不適切なオペコード（`0xEB` 等）が選択され、バイナリが破壊されます。

### 3.2. 単調増加（Sticky Size）の保証
マルチパス中、一度拡張された命令サイズは、その後のパスで**絶対に縮小させてはいけません**。サイズが縮小すると、後続のラベルアドレスが前方にずれ、別のジャンプが短縮可能になり、再び PC が変動するという「サイズ振動（Oscillation）」が発生し、アセンブルが収束しなくなる恐れがあるためです。

### 3.3. 前方参照ガード
Pass 1 の初期段階で、ターゲットラベルのアドレスが未確定（シンボルテーブルに未登録、または前回のパスの値しかない）場合、安易にサイズを拡張してはいけません。アドレスが未確定の間は最短（2バイト）を維持し、アドレスが確定してから拡張判定を行います。

### 3.4. セクション PC リセットの連動
COFF 形式などのセクション構造を持つ場合、`[SECTION]` 命令で PC を 0 にリセットする必要があります。BDO は PC とラベルアドレスの相対距離に依存するため、PC の管理が正確でないと（特に `ALIGNB` 等の疑似命令が混ざる場合）、1〜2バイトの微細なサイズ不一致が発生します。

## 4. 計算式

x86 の相対ジャンプの変位（Displacement）は、**「ジャンプ命令の次の命令の先頭アドレス」**からの距離です。

```
Displacement = TargetAddress - (JumpInstAddress + 2)
```
※ `+2` は、最短の `rel8` 命令のサイズ（オペコード 1バイト + 変位 1バイト）を基準とした固定値です。Pass 1 でサイズが変わっても、この基準を維持することで判定の安定性を高めています。

## 5. 既知の挙動

- `nask` や `nasm` と比較して、`rakusk` は極めて攻撃的に最適化を行います。そのため、一部のテストケースで `nask` よりもバイナリが数バイト小さくなることがありますが、これは正常な最適化の結果です。ただし、間接 FAR ジャンプの誤認など、セマンティクスを破壊する最適化はバグとして修正済みです。
