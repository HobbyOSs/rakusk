# COFF (WCOFF) 実装の詳細

`rakusk` における COFF 形式のバイナリ生成に関する詳細な仕様と、`nask`/`gosk` との互換性のための実装上の工夫を記録します。

## シンボルテーブルの構造

COFF シンボルテーブルは、各エントリ 18 バイトで構成されます。

### 登録順序
`nask` 互換のバイナリを生成するため、以下の順序でシンボルを登録します。

1.  `.file` シンボル (Idx 0) + 補助レコード (Idx 1)
2.  `.text` セクションシンボル (Idx 2) + 補助レコード (Idx 3)
3.  `.data` セクションシンボル (Idx 4) + 補助レコード (Idx 5)
4.  `.bss` セクションシンボル (Idx 6) + 補助レコード (Idx 7)
5.  `EXTERN` シンボル群 (Idx 8 〜)
6.  `GLOBAL` シンボル群

この順序（特に `EXTERN` が `GLOBAL` より前）にすることで、リロケーションテーブルから参照されるインデックスを期待値に合わせることができます。

### 補助レコード (Auxiliary Record)
セクションシンボルの直後の補助レコード（18バイト）には、以下の情報を含める必要があります。
- 最初の 4 バイト: セクションのサイズ
- 次の 2 バイト: そのセクションにおけるリロケーションの数

## 文字列テーブルの構造

8文字を超えるシンボル名は、文字列テーブル（シンボルテーブルの直後）に格納されます。

### 構築ルール
- **順序**: アセンブリソース内での **出現順序（定義順）** を厳密に維持する必要があります。
- **8文字制限**: 8文字以下の名前は、文字列テーブルに入れず、シンボルテーブルの名前フィールドに直接格納します。
- **重複**: 同一名のシンボルが複数回定義（GLOBALとラベル等）されても、文字列テーブル内では一意に管理します。

## リロケーション

`CALL` や `JMP` で `EXTERN` シンボルを参照する場合、リロケーションエントリを生成します。

- **型**: `20` (`REL_I386_REL32`) を使用。
- **インデックス**: シンボルテーブルにおける `EXTERN` シンボルの位置（0ベースのインデックス）を指定。

## 注意点
- オペランドを持たない 32 ビット命令（`IRETD`, `PUSHAD`, `POPAD` 等）を 32 ビットモードでアセンブルする際、`0x66` プレフィックスを付加してはいけません。これを誤るとセクションサイズがずれてしまい、後続のシンボルテーブル位置がすべて狂う原因となります。
